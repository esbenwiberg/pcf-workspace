01 Overview

The problem this architecture solves, and the philosophy behind it.

Building UI controls for Microsoft Power Apps (PCF components) and Dynamics 365 web resources is traditionally painful. The platform's APIs are clunky, the dev loop is slow (deploy → publish → refresh → squint at form), and testing is basically "click around in a browser and pray."

This workspace flips that on its head with one core principle:

PCF is a deployment target, not a development environment.

You write pure React components that know nothing about Power Apps. The PCF/web-resource layer is a thin shell that maps platform context to React props. That's it.

The result: you get modern React DX — hot reload, Storybook, Vitest, proper component isolation — while still deploying to Power Apps and Dynamics 365.

03 The Three-Layer Architecture

Each layer has exactly one job. Cross-layer imports are illegal.

🎯

Layer 1 — Apps

Pure React components. Import Fluent UI, shared hooks, and IDataverseClient. Never import PCF or Xrm types. Could run in any React host.

🔌

Layer 2 — Shells

Thin adapters. PCF shells map context.parameters to props. Web resource shells parse query params. Both inject the right Dataverse client. Under 100 lines.

📦

Layer 3 — Packages

Shared foundations. The Dataverse client abstraction, UI primitives, hooks, PCF context mocks, and test utilities. Consumed by all other layers.

04 The Dataverse Client Abstraction

One interface, three implementations. The linchpin of the whole architecture.

This is the key insight that makes everything work. Apps depend on an interface, not a concrete API. The right implementation gets injected depending on where the code runs.

packages/dataverse/src/client.interface.ts
export interface IDataverseClient {
  retrieveMultiple<T>(query: ODataQuery): Promise<PagedResult<T>>;
  retrieve<T>(entityType: string, id: string, select?: string[]): Promise<T>;
  create(entityType: string, data: Record<string, unknown>): Promise<string>;
  update(entityType: string, id: string, data: Record<string, unknown>): Promise<void>;
  delete(entityType: string, id: string): Promise<void>;
  getEntityMetadata(entityType: string): Promise<EntityMetadata>;
  getOptionSet(entityType: string, attribute: string): Promise<OptionSetValue[]>;
}

Three Implementations

Implementation Wraps Used In
MockDataverseClient In-memory store with OData filter simulation Storybook stories, unit tests
PcfDataverseClient context.webAPI (PCF framework) PCF control shells
XrmDataverseClient Xrm.WebApi (D365 global) Web resource shells

Why this matters: The app component is written once. The same AccountPicker renders in Storybook with mock data, in a PCF control on a Power Apps form, and inside a D365 web resource iframe. Zero conditional logic, zero platform checks inside the component.

MockDataverseClient — More Than a Stub

The mock client isn't a dumb pass-through. It simulates real OData behavior:

  • Filter parsing — Supports contains(field, 'value'), field eq 'value', compound and expressions
  • Latency simulation — Configurable delay to test loading states
  • CRUD operations — Create/update/delete mutate the in-memory store
  • Seeding — .seed() and .reset() for test setup/teardown

05 PCF & Web Resource Shells

Thin adapters that translate platform APIs to React props.

PCF Shell Pattern

pcf/sample-app/AccountPickerControl/index.tsx
export class AccountPickerControl
  implements ComponentFramework.StandardControl<IInputs, IOutputs> {

  init(context, notifyOutputChanged, _state, container) {
    this.root = createRoot(container);
    this.render(context);
  }

  private render(context) {
    this.root.render(
      <FluentProvider>
        <AccountPicker
          // Map PCF context → React props
          selectedAccountId={context.parameters.selectedAccountId?.raw}
          disabled={context.mode.isControlDisabled}
          dataverseClient={new PcfDataverseClient(context.webAPI)}
          onAccountSelected={(id) => {
            this.currentValue = id;
            this.notifyOutputChanged();
          }}
        />
      </FluentProvider>
    );
  }

  getOutputs() {
    return { selectedAccountId: this.currentValue };
  }
}

That's the entire PCF shell. 66 lines. All it does is:

  1. Mount React into the PCF container
  2. Map context.parameters → props
  3. Inject PcfDataverseClient
  4. Pipe callbacks back to notifyOutputChanged

Web Resource Shell Pattern

webresources/sample-app/src/index.tsx
// Parse D365's "data" query parameter
const config = parseDataParam();
const client = new XrmDataverseClient();

const root = createRoot(
  document.getElementById('root')
);

root.render(
  <FluentProvider theme={webLightTheme}>
    <AccountPicker
      selectedAccountId={config.selectedAccountId}
      filter={config.filter}
      dataverseClient={client}
      onAccountSelected={(id) => {
        // Notify parent frame
        window.parent?.postMessage(
          { type: 'webresource:accountSelected',
            accountId: id },
          '*'
        );
      }}
    />
  </FluentProvider>
);

Same idea, different platform glue:

  1. Parse D365 data query param (JSON or URL-encoded)
  2. Inject XrmDataverseClient (resolves Xrm.WebApi)
  3. Mount the exact same component
  4. Use postMessage for iframe communication

Build Pipeline

Both shell types use esbuild to produce a single IIFE bundle. A custom workspacePlugin resolves @workspace/* imports to TypeScript source files, so no pre-compilation step is needed for packages.

06 Storybook & Developer Experience

Storybook IS the development environment. Not an afterthought.

Since apps are pure React, Storybook becomes the primary dev loop. Custom decorators simulate the Power Apps host environment so components behave realistically without deploying anything.

Custom Decorators

withPowerAppsShell

Wraps components in the same DOM nesting Power Apps uses (.pa-form → .pa-section → .pa-cell → .customControl). Toolbar toggles between form, view, canvas, and bare modes. Also controls width.

withFormContext

Provides a mock PCF form context via React context. Components can access parameters, mode, formatting, navigation — all without a real PCF runtime.

withDatasetContext

Provides a mock PCF dataset context with records, columns, paging, sorting, and filtering. For dataset-type controls.

Story Example

Stories inject different client configurations to exercise every state:

apps/sample-app/src/AccountPicker.stories.tsx
// Normal data
const mockClient = new MockDataverseClient({ entities: { account: mockAccounts } });

// Empty state
const emptyClient = new MockDataverseClient();

// Slow network
const slowClient = new MockDataverseClient({ entities: { account: mockAccounts }, latency: 2000 });

// Broken API
const errorClient: IDataverseClient = {
  retrieveMultiple: () => Promise.reject(new Error('Network error')),
  // ...
};

// Stories: Default, WithSelection, Disabled, EmptyResults, SlowLoading, ErrorState

Six stories, six states. Default, pre-selected, disabled, empty, loading, and error. Every edge case is visually testable without touching a Dataverse environment. Ship bugs? Not today.

07 Testing Strategy

A layered approach — fast feedback first, visual validation second.

Level Tool What It Tests Speed
1. Type Check pnpm typecheck Cross-package type safety via project references ~2s
2. Unit Tests pnpm test (Vitest) Component logic with MockDataverseClient + jsdom ~5s
3. Visual / E2E Playwright + Storybook Rendered output, interaction flows ~30s
4. Integration Manual / pcf-dev-proxy Real Dataverse, real PCF runtime ~60s

The key: levels 1-3 require zero Dataverse connectivity. You can develop and validate entirely offline. The MockDataverseClient makes this possible by simulating real OData behavior including filtering, pagination, and error states.

Test helper from packages/test-utils
// renderWithProviders wraps in FluentProvider automatically
import { renderWithProviders, mockAccounts } from '@workspace/test-utils';

it('renders account list', async () => {
  const client = new MockDataverseClient({ entities: { account: mockAccounts } });
  renderWithProviders(<AccountPicker dataverseClient={client} />);
  // assert away...
});

08 Scaffolding & Code Generation

One command to go from zero to a fully wired control.

The tooling/scaffold-app.mjs generator creates all the boilerplate with correct conventions, package references, and build configs.

App Only

pnpm new:app my-control

Creates: component, types, index, stories, tests, package.json, tsconfig.json

App + PCF Shell

pnpm new:app my-control --pcf

Also creates: PCF shell, ControlManifest.Input.xml, esbuild config

App + Web Resource

pnpm new:app my-control --webresource

Also creates: WR shell, index.html host, esbuild config

Full Stack

pnpm new:app my-control --pcf --webresource

The whole enchilada. App + both deployment shells, ready to go.

09 Why This Design Is Good

It's not clever for the sake of it. Every decision solves a real pain point.

⚡

10x Faster Dev Loop

Traditional PCF dev: edit → build → deploy → publish → refresh form → navigate to field → see result. This workspace: edit → Storybook hot reloads. The feedback cycle drops from minutes to milliseconds.

🧪

Actually Testable

PCF components are notoriously hard to test because they depend on the ComponentFramework runtime. By moving all logic to pure React, everything is testable with standard tools (Vitest, Testing Library, Playwright). The MockDataverseClient eliminates the need for a live Dataverse instance.

🔄

Write Once, Deploy Anywhere

The same AccountPicker component runs as a PCF control and a web resource without a single line of conditional code. Need a third deployment target? Write a 50-line shell. The app doesn't change.

🛡

Enforced Boundaries

The monorepo structure makes it physically impossible for apps to depend on PCF types. TypeScript project references enforce the dependency graph at compile time. Violations fail the build, not the code review.

🏗

Low Ceremony Scaling

Adding a new control is one command: pnpm new:app foo --pcf. The scaffold tool generates everything with correct conventions. No copy-paste, no missed configuration, no "oh wait, I forgot the build config." Teams scale without the bus factor.

🔧

Dependency Injection Done Right

The IDataverseClient interface is a textbook application of the Dependency Inversion Principle. High-level modules (apps) don't depend on low-level modules (PCF/Xrm APIs). Both depend on abstractions. This isn't academic — it's what makes the whole test and multi-target story work.

The Alternative (and Why It Sucks)

Without this architecture, a typical PCF project looks like:

  • Business logic tangled with context.parameters and notifyOutputChanged
  • No Storybook because the component can't render outside PCF
  • No unit tests because everything needs the framework runtime
  • Every new control is a copy-paste-modify of the last one
  • Web resource versions require rewriting the component from scratch
  • Dev cycle measured in minutes, not milliseconds

The bottom line: This architecture treats the Power Apps platform as a delivery mechanism, not a development framework. Your components are portable, testable, and fast to iterate on. The platform-specific code is confined to 50-100 line adapters that are trivial to write, review, and maintain.

Trade-offs — Nothing Is Free

In the interest of honesty, here's what you're paying for this setup:

Trade-off Mitigation
Initial setup complexity (monorepo tooling, workspace config, esbuild plugins) One-time cost, amortized across all controls. Scaffold tool eliminates ongoing friction.
Extra abstraction layer (IDataverseClient) to maintain The interface is small and stable. Dataverse's CRUD API rarely changes.
Mock client may diverge from real API behavior Integration test layer (Level 4) catches this. Mock covers 90% of dev-time needs.
Bundle size slightly larger (Fluent UI + React in every control) PCF controls are loaded per-form, not globally. A few hundred KB is fine for enterprise.
